## Part 1 – Decide What Goes into Cosmos DB

Since my project includes both a relational database (Azure SQL) and Cosmos DB (NoSQL), I have split the data as follows:

### Data that remains in SQL (relational database)
- **PredictionRecord** model (id, name, sex, age, pclass, fare, sibsp, parch, embarked, family_size, age_group, survived_prediction, probability, rating, created_at)
- All structured, relational data that is frequently queried with filters, sorting, and pagination (e.g., prediction history list).
- Data that requires **ACID transactions** (e.g., saving a prediction and updating rating in the same request).
- Data that is used by Django’s admin interface and session management.

### Data that is stored (or duplicated) in Cosmos DB
- A **denormalised copy** of each prediction record, stored as a JSON document.
- The document includes: id, userId, passengerName, prediction (Survived/Perished), probability, inputData (pclass, sex, age, etc.), timestamp, and rating (when updated).

### Justification

**Why move this data to Cosmos DB (NoSQL)?**

1. **Flexible schema** – Cosmos DB allows me to store additional fields (e.g., `userId`, `passengerName`) without migrating the relational table. In the future, I can add new fields like `deviceInfo`, `location`, or `sessionId` without any schema changes.

2. **Read‑heavy analytics** – The `/cosmos-history/` page demonstrates a simple read operation. For real‑time analytics or dashboards, Cosmos DB’s low‑latency reads and global distribution would be beneficial.

3. **Scalability** – If the application grows to millions of predictions, Cosmos DB can scale horizontally with automatic partitioning, while SQL scaling is more expensive and complex.

4. **Event sourcing / activity logs** – Each prediction can be treated as an immutable event. Cosmos DB is excellent for storing event streams and time‑series data.

5. **Separation of concerns** – The relational database remains the source of truth for transactional operations, while Cosmos DB serves as a secondary store for read‑optimised views, logs, or integration with other services.

**Why not keep everything in SQL?**

- SQL is not ideal for storing semi‑structured or evolving data (e.g., additional metadata per prediction).
- Writing to SQL is fine, but reading large volumes of historical data for analytics can be slow and expensive.
- Cosmos DB offers native JSON support, automatic indexing, and global replication, which would be difficult to achieve with SQL alone.

## Part 2 – Architecture Design

### How the system works (both SQL and Cosmos DB)

My Django application uses **two databases side by side**:

- **Azure SQL Database** – primary transactional store (relational).
- **Azure Cosmos DB** – secondary document store (NoSQL).

### Data flow

1. **User submits a prediction form** (`PredictionFormView`):
   - Data is validated by Django form.
   - Prediction is calculated using the ML model.
   - The result is saved to **SQL** (`PredictionRecord` table).
   - Immediately after, the same data is converted to a JSON document and written to **Cosmos DB** (container `predictions`, partition key `userId` → fixed as `"titanic_app"`).
   - If Cosmos DB write fails, the error is logged but the user is not interrupted.

2. **User views prediction history** (`/predictions/`):
   - Reads data exclusively from **SQL** (ordered by date, paginated).
   - This is the main UI for users to see their past predictions.

3. **User rates a prediction** (`submit_rating`):
   - Rating is saved to **SQL** first (updating the `rating` column).
   - Then, the same rating is updated in the corresponding Cosmos DB document (using the same partition key and document id).
   - If Cosmos DB update fails, only SQL is updated; consistency is logged.

4. **User views Cosmos DB history** (`/cosmos-history/`):
   - Reads all documents from **Cosmos DB** (using a SQL‑like query).
   - This page demonstrates the NoSQL read path and shows the same data (including ratings) as a separate view.

### When SQL is used
- All write operations (create, update) for the core `PredictionRecord` model.
- Read operations for the main history list (ordering, pagination, filtering).
- Admin interface and user session management.

### When Cosmos DB is used
- Secondary write (duplicate) of each prediction document.
- Update of rating in the document.
- Read operations for the dedicated `/cosmos-history/` page.
- Future use: real‑time analytics, global read replicas, or integration with Azure Functions.

### Diagram (text description)

```
User → Prediction Form → Django View
                              ↓
                         Azure SQL (primary)
                              ↓
                         (if success) → Cosmos DB (secondary)

User → Rating Form → Django View
                        ↓
                   Azure SQL (update rating)
                        ↓
                   (if success) → Cosmos DB (update rating)

User → History Page → Django View → Azure SQL → HTML
User → Cosmos History Page → Django View → Cosmos DB → HTML
```

This architecture allows me to keep the benefits of a relational database for transactional integrity while exploring the advantages of a NoSQL document store for read‑heavy, schema‑flexible scenarios.

## Part 5 – Handle Challenges

### 1. What happens if Cosmos DB is unavailable?

In my implementation, all Cosmos DB operations (write after prediction, update rating) are wrapped in `try/except` blocks. If Cosmos DB is unavailable or an error occurs, the exception is caught, logged to the console/logger, and **the main application flow continues unaffected**. The user still receives their prediction result and can submit ratings; the only consequence is that the data is not written to Cosmos DB. This design ensures that the core functionality (survival prediction) is never blocked by an optional secondary service.

**Example from code:**
```python
try:
    cosmos = CosmosService()
    cosmos.create_item(cosmos_item)
except Exception as e:
    logger.error(f"Cosmos DB error: {e}")
    # No re-raise – the user is not interrupted
```

If Cosmos DB is unavailable for reading (`cosmos_history` page), the view will either show an empty list or raise an exception – but since this is a separate read‑only page, it does not affect the main prediction workflow.

---

### 2. How do you validate data before saving?

Before saving to both SQL and Cosmos DB, I rely on **Django’s built‑in form validation** (`PredictionForm`). The form checks that required fields are present, data types are correct (e.g., age as integer, fare as decimal), and any custom validation logic (e.g., age range) is applied. This ensures that only valid data enters the system.

Additionally, before writing to Cosmos DB, I manually convert certain fields (e.g., `fare` from `Decimal` to `float`) to ensure JSON serialisation works. I also check for `None` values and provide defaults where appropriate.

**Key validation points:**
- Django Form `is_valid()` – prevents invalid data from reaching the database.
- Manual type conversion – ensures JSON compatibility for Cosmos DB.
- Null handling – e.g., `fare` default to `0.0` if missing.

---

### 3. How do you handle consistency between SQL and NoSQL?

My system uses **eventual consistency** with a **SQL‑first** strategy. The authoritative source of truth is the **Azure SQL Database** (or SQLite during development). Cosmos DB is treated as a secondary, denormalised copy for read‑only display and future scalability experiments.

**Consistency approach:**
- **Write order:** First, data is saved to SQL. Only after successful SQL commit, the same data is written to Cosmos DB. This guarantees that the primary store always has the latest information.
- **Failure handling:** If Cosmos DB write fails, SQL data remains unchanged, and the inconsistency is logged. No automatic retry or back‑sync is implemented (for this lab, it is acceptable because Cosmos DB is not critical).
- **Reads:** The main application reads from SQL. The `/cosmos-history/` page reads from Cosmos DB separately – it may temporarily show missing or outdated data if Cosmos DB write failed earlier.
- **Updates (rating):** When a user submits a rating, the SQL record is updated first. Then, using the same fixed partition key (`"titanic_app"`), the corresponding Cosmos DB document is updated. If the Cosmos DB update fails, the rating remains correct in SQL, and the inconsistency is logged.

**In a production system,** I would implement a background queue (e.g., using Celery or Azure Service Bus) to retry failed Cosmos DB writes, or use Cosmos DB’s change feed to sync from SQL. However, for this lab, the current approach clearly demonstrates the trade‑offs and is sufficient to meet the requirements.

## Part 6 – Reflection Answers

**1. What worked well with Cosmos DB?**

The integration was smooth once the connection string and keys were set up. The `azure-cosmos` Python SDK is well documented, and the `CosmosClient` works intuitively. Writing and reading JSON documents is straightforward, and the ability to query using SQL-like syntax made it easy to adapt. The Data Explorer in Azure portal is very helpful for debugging.

**2. What was difficult?**

The most challenging part was handling data type conversions – specifically, the `Decimal` type from SQL Server cannot be directly serialised to JSON. I had to explicitly convert `fare` to `float` before writing to Cosmos DB. Another difficulty was choosing and consistently using the partition key. Initially I used `userId` (which could be empty), causing errors when updating documents. I finally settled on a fixed partition key `"titanic_app"` to simplify the process.

**3. When would you NOT use Cosmos DB?**

I would not use Cosmos DB for:
- Small applications with low traffic and simple data needs (SQLite or a traditional SQL database would be cheaper and easier).
- Applications that require complex multi‑table joins or transactions across many entities (relational databases are better suited).
- Projects with very tight budgets and predictable, low‑volume workloads – the serverless Cosmos DB tier is affordable, but a single SQL database may be even cheaper.

**4. How would your system behave at scale?**

With millions of users, Cosmos DB would automatically scale horizontally using partitions. My chosen fixed partition key (`"titanic_app"`) would become a **hot partition** because all documents would go to the same logical partition, limiting throughput. That design is not scalable. In production, I would use a more distributed partition key, such as `userId` (if users are logged in) or a random suffix to spread writes evenly. The relational database (Azure SQL) would also need scaling (e.g., using Hyperscale tier or read replicas), but it would likely become a bottleneck earlier than Cosmos DB.

**5. Would you redesign anything?**

Yes. I would:
- Use a **dynamic partition key** (e.g., `userId` or a hash of `id`) to avoid hot partitions.
- Move the Cosmos DB credentials to Azure Key Vault instead of `.env` for better security.
- Implement a **retry policy** with exponential backoff for transient failures.
- Possibly use **Change Feed** to sync data from Cosmos DB back to SQL or trigger serverless functions.

## 数据流描述（SQLite/Azure SQL 与 Cosmos DB）

在我的泰坦尼克号生存预测 Django 应用中，我使用 **Azure SQL 数据库** 作为结构化预测记录（乘客详情、生存预测、概率、评分等）的主要关系型数据库。应用的主要数据都存储在这里，`PredictionRecord` 模型也持久化存储于此。

**Cosmos DB** 与 SQL 并行使用，作为用于存储相同预测数据的辅助、灵活的文档型数据库。每次用户提交预测表单时，应用都会将包含相同关键信息的 JSON 文档写入 Cosmos DB（位于 `predictions` 容器中）。这使我能够尝试使用 NoSQL 数据库，并在之后扩展实时分析或全局复制等功能，而无需修改核心关系模式。

当用户对预测进行评分（1-5 星）时，评分首先保存到 SQL 数据库（更新 `PredictionRecord` 中的 `rating` 列）。然后，通过使用相同的固定分区键（`"titanic_app"`）更新相应的文档，将相同的评分推送到 Cosmos DB。Cosmos DB 历史记录页面（`/cosmos-history/`）会从 Cosmos DB 读取所有文档并显示它们，包括评分（如果有）。

**使用 SQL 时：**

- 主要数据持久化

- 复杂查询（过滤、排序、分页）

- 关系（例如，用户会话、管理界面）

**使用 Cosmos DB 时：**

- 用于相同预测的辅助文档存储

- NoSQL 集成演示

- 未来用途：快速全局读取、灵活的模式日志记录、离线同步

## 第 6 部分 – 反思答案

**1. Cosmos DB 的哪些方面表现出色？**

一旦设置好连接字符串和键，集成过程就非常顺利。 `azure-cosmos` Python SDK 文档齐全，`CosmosClient` 的使用也非常直观。读写 JSON 文档非常简单，而且能够使用类似 SQL 的语法进行查询，这使得上手非常容易。Azure 门户中的数据资源管理器对调试非常有帮助。

**2. 遇到的困难是什么？**

最具挑战性的部分是处理数据类型转换——具体来说，SQL Server 中的 `Decimal` 类型无法直接序列化为 JSON。我必须在写入 Cosmos DB 之前显式地将 `fare` 转换为 `float`。另一个困难是选择并始终如一地使用分区键。最初我使用了 `userId`（它可以为空），这导致在更新文档时出现错误。最终，我决定使用固定的分区键 `"titanic_app"` 来简化流程。

**3. 什么情况下不建议使用 Cosmos DB？**

以下情况不建议使用 Cosmos DB：

- 流量较低、数据需求简单的应用程序（SQLite 或传统的 SQL 数据库更便宜也更容易）。

- 需要复杂多表连接或跨多个实体的事务的应用程序（关系型数据库更适合）。

- 预算非常紧张且工作负载可预测、低容量的项目——无服务器 Cosmos DB 层价格实惠，但单个 SQL 数据库可能更便宜。

**4. 您的系统在大规模部署时会如何运行？**

面对数百万用户，Cosmos DB 会使用分区自动进行横向扩展。我选择的固定分区键（`"titanic_app"`）会成为**热分区**，因为所有文档都会写入同一个逻辑分区，从而限制吞吐量。这种设计不具备可扩展性。在生产环境中，我会使用更分散的分区键，例如 `userId`（如果用户已登录）或随机后缀，以均匀分散写入。关系型数据库（Azure SQL）也需要扩展（例如，使用超大规模层或只读副本），但它可能比 Cosmos DB 更早成为瓶颈。

**5.你会重新设计什么吗？

是的。我会：

- 使用**动态分区键**（例如，`userId` 或 `id` 的哈希值）来避免出现热分区。

- 将 Cosmos DB 凭据从 `.env` 文件移至 Azure Key Vault，以提高安全性。

- 针对瞬态故障实施**指数退避重试策略**。

- 可能使用**变更源**将数据从 Cosmos DB 同步回 SQL 或触发无服务器函数。
